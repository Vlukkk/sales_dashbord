# Database Ingestion Plan

Текущий целевой принцип:

- `data/*.xml` — сырой источник продаж.
- `SKU-Lieferant.xlsx` — master data по поставщикам и SKU.
- `Lagerbestand...txt` — snapshot остатков FBA.

## 1. Продажи

Правило на сейчас:

- рабочий источник продаж берём из папки `data/`
- если файлов несколько, для JSON/локальной сборки используем последний по имени `product - *.xml`
- старый root-level XML оставлен только как fallback

Почему так:

- это безопасно для текущего дашборда
- не ломает deploy-скрипт
- даёт понятную точку входа для дальнейшего импорта в PostgreSQL

Открытое решение:

- если в `data/` будут лежать периодические полные выгрузки, их нельзя просто append-ить в `sales`
- в этом случае нужен либо:
  - один актуальный consolidated XML для runtime
  - либо staging-таблица + merge/upsert по стабильному business key

## 2. Поставщики

`SKU-Lieferant.xlsx` считаем master data:

- файл меняется редко
- это dimension / mapping layer, а не поток событий
- загружается в `suppliers` + `sku_supplier`

Практический режим:

- при каждом импорте можно делать upsert
- позже можно оптимизировать по hash файла и пропускать неизменившийся import

## 3. Остатки FBA

Остатки должны храниться как snapshots:

- каждая загрузка = отдельная дата снимка
- таблица `inventory_snapshots` уже под это подготовлена

Это важно, потому что для остатков почти всегда нужна динамика:

- что было сегодня
- что было неделю назад
- какие SKU выпали в ноль

## 4. Будущая автоматизация Amazon

Правильный целевой вариант:

- не писать в проект руками TXT-файл
- забирать FBA inventory по API Amazon и складывать raw-файлы/ответы в отдельный ingestion pipeline

Рекомендованный подход:

- короткий путь: scheduled job получает inventory report от Amazon и пишет raw snapshot в storage
- более зрелый путь: отдельный importer-service, который:
  - тянет данные из Amazon
  - сохраняет raw payload
  - пишет запись в `data_imports`
  - импортирует snapshot в PostgreSQL

Почему так лучше:

- появляется трассируемость
- можно повторно проигрывать импорт
- нет ручной подмены файлов на сервере

## 5. Следующий практический шаг

Следующий разумный шаг для проекта:

1. закрепить локальный PostgreSQL запуск
2. прогнать первый импорт в БД
3. проверить `/api/dashboard/bootstrap`
4. открыть фронт с `VITE_DATA_SOURCE=api`
5. после этого решать, как именно разруливать историю продаж из нескольких XML
